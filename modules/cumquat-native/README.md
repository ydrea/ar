# Cumquat Native Engine

`@cumquat/native-engine` is the stateful C++ projection engine used by the AR view.

The canonical native implementation lives in:

```text
modules/cumquat-native/cpp/
```

Android and iOS compile the same C++20 sources. Platform-specific files only register and package the shared implementation for React Native.

## Status

- React Native New Architecture TurboModule
- Shared C++20 implementation
- Android registration through CMake and `OnLoad.cpp`
- iOS registration through CocoaPods and an Objective-C++ module provider
- React Native Codegen contract in TypeScript
- Typed TypeScript wrapper for application code
- Android and iOS integration generated by the Expo config plugin
- App-local/private package at the moment

## Architecture

```text
ARBetaNativeOverlayView.tsx
        |
        | typed TypeScript API
        v
src/CumquatEngine.ts
        |
        | TurboModuleRegistry
        v
specs/NativeCumquat.ts
        |
        | React Native Codegen / JSI
        v
cpp/bridge/NativeCumquatModule.cpp
        |
        | owns native engine handles
        v
cpp/core/Engine.cpp
        |
        +--> cpp/geo/Geodesy.cpp
        |
        +--> cpp/projection/Projection.cpp
```

Platform registration:

```text
Android                              iOS
-------                              ---
CMakeLists.txt                       CumquatNative.podspec
OnLoad.cpp                           NativeCumquatModuleProvider.mm
        \                            /
         +--> NativeCumquatModule <--+
```

The runtime flow is:

1. Create an engine instance.
2. Initialize it with the POI dataset.
3. Update mutable view state when gestures change FOV or distance limits.
4. Feed sensor snapshots to `update()`.
5. Read the latest frame with `getFrame()`.
6. Optionally hit-test visible POIs with `pick()`.
7. Dispose the engine when the screen unmounts.

### `specs/NativeCumquat.ts`

The React Native Codegen contract defines the low-level TurboModule methods:

```text
getVersion
createEngine
initialize
setViewState
update
getFrame
pick
destroyEngine
```

The spec intentionally uses Codegen-compatible object types. Application-facing types are defined separately in `src/types.ts`.

### `src/CumquatEngine.ts`

The typed JavaScript wrapper:

- creates and stores a native engine handle;
- exposes the public API as a class;
- prevents calls after disposal;
- keeps raw TurboModule calls out of UI code.

### `cpp/bridge/NativeCumquatModule.*`

The cross-platform JSI/TurboModule bridge:

- converts JavaScript objects and arrays into C++ structures;
- stores multiple engine instances in a handle map;
- validates engine handles;
- serializes frame data across the JSI boundary;
- protects the engine registry with a mutex.

Transport and validation belong here. Projection policy does not.

### `cpp/core/Engine.*`

The central stateful engine owns:

- immutable engine configuration;
- the initialized POI dataset;
- mutable view state;
- the most recent frame snapshot;
- visible-POI picking.

`initialize()` normally runs once per engine instance. Gesture changes should call `setViewState()` rather than recreate or reinitialize the engine.

### `cpp/core/Types.h`

The canonical native data model contains:

- `GeoPoint`
- `Quaternion`
- `POI`
- `EngineConfig`
- `ViewState`
- `SensorState`
- `VisiblePOI`
- `FrameSnapshot`
- `PickResult`

Keep this contract synchronized conceptually with `src/types.ts`.

### `cpp/geo/Geodesy.*`

Geographic calculations:

- geodetic point to ECEF conversion;
- ECEF to local ENU conversion;
- initial bearing calculation.

The local world coordinate system is:

```text
+x = east
+y = north
+z = up
```

### `cpp/projection/Projection.*`

Camera transformation and screen projection.

Two orientation paths are supported:

1. Quaternion mode, used by the current JavaScript sensor pipeline.
2. Heading/pitch/roll mode, retained for conventional native callers and tests.

Quaternion mode mirrors the JavaScript `q * v * q^-1` behavior. It also calculates directional screen coordinates for behind-camera POIs so the UI can place stable edge indicators while `visible` remains false.

## Public TypeScript API

```ts
import {CumquatEngine} from "@/modules/cumquat-native/src";
import type {
  POIInput,
  SensorState,
  ViewState,
} from "@/modules/cumquat-native/src";
```

### Create

```ts
const engine = CumquatEngine.create({
  datasetRadiusMeters: 135_000,
  maxVisiblePOIs: 256,
});
```

`datasetRadiusMeters` is a hard, immutable spatial boundary. POIs outside it are not projected and do not cross the JSI boundary.

`maxVisiblePOIs` caps the visible-only list. It does not cap `projectedPOIs` for POIs inside the hard radius.

### Initialize

```ts
engine.initialize([
  {
    id: "26",
    name: "Sljeme",
    latitude: 45.89946265300375,
    longitude: 15.94482091926767,
    altitude: 1033,
  },
]);
```

POI ordering matters. A native `poiIndex` refers to the index in this initialized array.

### Set mutable view state

```ts
const viewState: ViewState = {
  horizontalFovDegrees: 90,
  minDistanceMeters: 0,
  maxDistanceMeters: 13_500,
};

engine.setViewState(viewState);
```

Native validation requires:

- horizontal FOV greater than `1` and less than `179` degrees;
- minimum distance greater than or equal to `0`;
- maximum distance strictly greater than minimum distance;
- finite numeric values.

### Update from sensors

```ts
const sensorState: SensorState = {
  timestampNs: Date.now() * 1_000_000,
  location: {
    latitude: 45.80041,
    longitude: 15.95619,
    altitude: 120,
  },
  orientationQuaternion: {
    x: 0,
    y: 0,
    z: 0,
    w: 1,
  },
  headingDegrees: 0,
  pitchDegrees: 0,
  rollDegrees: 0,
  viewportWidth: 1000,
  viewportHeight: 600,
};

const sequence = engine.update(sensorState);
```

When `orientationQuaternion` is present, quaternion mode is used. Otherwise, the heading/pitch/roll path is used.

### Read a frame

```ts
const frame = engine.getFrame();
```

A frame contains:

```ts
type FrameSnapshot = {
  sequence: number;
  timestampNs: number;
  projectedPOIs: readonly ProjectedPOI[];
  visiblePOIs: readonly ProjectedPOI[];
};
```

`projectedPOIs`:

- preserves stable POI-index order;
- includes each POI inside `datasetRadiusMeters`;
- includes visible, offscreen, behind-camera and distance-clipped entries;
- supports labels and edge indicators without a second JavaScript projection pass.

`visiblePOIs`:

- contains only currently visible entries;
- is capped by `maxVisiblePOIs`;
- is sorted far-to-near by depth;
- is used by native picking.

Distance clipping is reported separately:

```text
clippedByDistance = "min" | "max" | null
```

### Pick and dispose

```ts
const result = engine.pick(screenX, screenY, 32);
engine.dispose();
```

Picking searches the current visible frame. Disposal is idempotent, and the TypeScript wrapper rejects later calls.

## Android build integration

`plugins/withCumquatNative.js` performs the Android integration during Expo prebuild:

1. Adds `externalNativeBuild` to `android/app/build.gradle`.
2. Generates `android/app/src/main/jni/OnLoad.cpp`.
3. Registers `NativeCumquatModule` in the C++ module provider.
4. Generates `android/app/src/main/jni/CMakeLists.txt`.
5. Points CMake to `modules/cumquat-native/cpp/`.
6. Enables C++20.

The Android target compiles:

```text
cpp/bridge/NativeCumquatModule.cpp
cpp/core/Engine.cpp
cpp/geo/Geodesy.cpp
cpp/projection/Projection.cpp
```

## iOS build integration

The iOS path uses the standard React Native pure-C++ TurboModule provider model.

### Codegen provider registration

The root `package.json` associates the JavaScript module name with its Objective-C++ provider:

```json
{
  "codegenConfig": {
    "ios": {
      "modulesProvider": {
        "NativeCumquat": "NativeCumquatModuleProvider"
      }
    }
  }
}
```

During CocoaPods installation, React Native Codegen uses that mapping to register the provider.

### Objective-C++ provider

```text
modules/cumquat-native/ios/
├── NativeCumquatModuleProvider.h
└── NativeCumquatModuleProvider.mm
```

The provider conforms to `RCTModuleProvider` and constructs the same `facebook::react::NativeCumquatModule` used on Android.

Objective-C++ is required because the provider interacts with both Objective-C protocols and C++ `std::shared_ptr` types.

### CocoaPods target

```text
modules/cumquat-native/CumquatNative.podspec
```

The podspec:

- compiles all shared C++ and iOS provider sources;
- exposes the shared headers;
- enables C++20;
- uses React Native's `install_modules_dependencies` helper;
- targets iOS 15.1 and later, matching Expo SDK 56.

The Expo config plugin adds this local pod inside the generated application target:

```ruby
pod 'CumquatNative', :path => '../modules/cumquat-native'
```

Do not manually add the C++ folder to Xcode. The CocoaPods target owns those source files.

## Native changes and EAS Update

Changes to any of these require a new native build:

```text
modules/cumquat-native/cpp/**
modules/cumquat-native/ios/**
modules/cumquat-native/specs/**
modules/cumquat-native/CumquatNative.podspec
plugins/withCumquatNative.js
package.json codegenConfig
native dependencies
Android or iOS build configuration
```

They cannot be delivered safely through `eas update`. OTA updates replace JavaScript and bundled assets, not compiled C++, CocoaPods targets, CMake targets or TurboModule registration.

Pure TypeScript/UI changes can use EAS Update only when they remain compatible with the installed native runtime.

## Performance model

The engine is designed to:

- initialize the dataset once;
- keep projection state in C++;
- apply the immutable hard radius before projection and serialization;
- update FOV and gesture distance limits without recreating the engine;
- return both full projected state and a visible-only subset;
- avoid a second JavaScript projection pass;
- let React commit only meaningful frame changes.

The hard radius is separate from gesture-controlled distance limits:

```text
datasetRadiusMeters
    immutable native query boundary

minDistanceMeters / maxDistanceMeters
    mutable presentation and gesture clipping range
```

A POI beyond the hard radius does not enter a frame. A POI inside the radius but outside the gesture range remains in `projectedPOIs` with `clippedByDistance` set, allowing directional edge indicators.

## Troubleshooting

### `NativeCumquat` is not found at runtime

Check that:

- the app was rebuilt after native changes;
- `./plugins/withCumquatNative` is present in `app.config.ts`;
- React Native Codegen ran successfully;
- the root `package.json` contains the iOS `modulesProvider` mapping;
- iOS `Podfile` contains the `CumquatNative` local pod;
- Android `OnLoad.cpp` includes and returns `NativeCumquatModule`;
- the installed binary is not an older build.

### iOS cannot find `NativeCumquatSpecJSI.h`

1. Delete the generated iOS project.
2. Run clean iOS prebuild on macOS.
3. Confirm that CocoaPods completed without error.
4. Confirm that `package.json` still points `jsSrcsDir` to `modules/cumquat-native/specs`.
5. Inspect `ios/build/generated/ios/` for the generated spec.

Do not copy generated headers into the repository.

### CocoaPods cannot find `CumquatNative`

Check:

```bash
ls modules/cumquat-native/CumquatNative.podspec
grep -n "CumquatNative" ios/Podfile
```

Fix the canonical podspec or config plugin, not the generated Pods project.

### Changes disappear after `prebuild --clean`

The edited file was generated. Move the change into the config plugin, podspec, provider, Codegen config or canonical C++ tree.

### JavaScript changed but native behavior did not

The device may still have an older binary. C++ and registration changes require a native rebuild and reinstall; an EAS Update is insufficient.

### Invalid POI index

`poiIndex` refers to the original array passed to `initialize()`. Keep that dataset stable for the lifetime of the engine and map frame entries against the same array.

## Maintenance rules

- Edit canonical C++ only under `modules/cumquat-native/cpp/`.
- Keep native and TypeScript types synchronized.
- Keep the bridge focused on transport and validation.
- Do not recreate the engine for ordinary view-state changes.
- Dispose every engine instance on unmount.
- Rebuild after native, podspec, provider or Codegen changes.
- Treat generated Android, iOS, Pods and Codegen files as disposable.
- Keep `plugins/withCumquatNative.js` idempotent because prebuild may run repeatedly.
- Validate both Android and iOS before extracting the module into a standalone package.
